# Cash Handling & Reception Monitoring Module — Use Case Summary

## What This Module Is

This module uses live or recorded CCTV footage to automatically monitor cash transactions at hotel reception desks, front offices, and payment counters. It is designed for hospitality and retail environments where tracking how cash is handled — who picks it up, where it goes, and whether it reaches the register — has direct value for loss prevention and operational transparency. No human needs to watch the camera feed; the system watches it and reports automatically.

## Scope

The module sits between a camera and a dashboard. It takes in video, processes it frame by frame, detects cash objects and the people handling them, tracks the lifecycle of each cash interaction, and flags suspicious behaviour. That data is written to a file and will eventually be pushed live to a dashboard over a web connection.

## What is Completed

| Feature | Description |
|---|---|
| **Cash Detection** | The system can identify banknotes and cash objects from a camera feed using a custom-trained YOLOv8 model. It works on both recorded video files and live camera streams. It filters out noise by rejecting detections that are too small, too large, or have extreme aspect ratios — only objects that look geometrically plausible as cash pass through. |
| **Context-Aware Filtering** | Standard object detection models sometimes flag random objects — signs, monitors, printed images — as cash. The system addresses this by validating every raw detection against real-world context. Cash is only accepted if it appears near a person's hands, on a designated counter or register zone, or in the gap between two people who are close enough to be exchanging items. Detections outside all three contexts are silently rejected. |
| **Cash-to-Person Association** | Once cash is detected, the system determines who is holding it. It uses a combination of bounding box overlap, center-point containment, and hand proximity scoring to assign each piece of cash to the nearest tracked person. Cash that is sitting on a counter with no person nearby is marked as unassigned and tracked separately. |
| **Role Classification** | Every tracked person is automatically classified as either a Cashier or a Customer based on where they spend their time. If a person stays inside a designated cashier zone for more than 60% of their tracked frames, they are labelled as staff. Everyone else defaults to Customer. No manual tagging is needed — the system learns roles from zone occupancy alone. |
| **Hand Detection & Interaction Tracking** | The system uses a YOLOv8-pose model to detect wrist positions for every person in the frame. When a customer's hand and a cashier's hand are within 90 pixels of each other inside a designated transaction zone, the system records it as a hand interaction event. These interactions are tracked frame by frame and used to infer when a cash exchange is happening. |
| **Cash State Tracking** | For every person in the frame, the system maintains a cash holding state — either NO_CASH or HOLDING_CASH. When cash appears near a person for several consecutive frames, a CASH_PICKUP event is generated. When cash disappears, the system determines whether it was deposited at a register, handed to another person, or pocketed. Each state transition is debounced to prevent flickering from detection noise — cash must be present for at least 8 frames to trigger a pickup, and absent for at least 15 frames to trigger a deposit or pocket event. |
| **Cash Exchange Detection** | The interaction analyzer fuses multiple signals — hand proximity, customer presence near the counter, and cash detection — over a rolling time window to determine when a cash exchange has taken place. If customer and cashier hands interact for at least 25 frames (one second) inside a transaction zone while cash is visible, a high-confidence exchange event is raised. Even without explicit cash detection, a prolonged hand interaction lasting 1.5 seconds or more triggers an inferred exchange at lower confidence. |
| **Fraud Detection** | The system evaluates business rules over the timeline of interactions and cash events to flag suspicious behaviour. Three specific rules are enforced. First, if a cashier pockets cash shortly after a customer exchange, a POSSIBLE_POCKETING alert is raised. Second, if a cashier handles cash at any time outside a transaction and moves it to their person, a CASH_POCKETED alert is raised. Third, if a cash exchange occurs but the cashier's hand never visits the register within the configured timeout (default 10 seconds), an UNREGISTERED_CASH alert is raised — indicating the money may not have been recorded. |
| **Zone-Based Monitoring** | Specific areas — cashier counters, cash registers, money exchange points — can be drawn directly onto the camera frame using a built-in annotation tool. Once defined, each zone is used to validate cash context, classify person roles, and gate interaction detection. The tool lets a non-technical operator define zones by clicking on the screen and no coordinates need to be entered manually. |
| **Data Logging** | All analytics events — cash pickups, deposits, handovers, pocketing, exchanges, and fraud alerts — are written to a structured data file. Each event records the event type, person ID, timestamp, frame index, zone, confidence score, and any associated partner. This file can be opened in Excel or used by any reporting tool. |
| **Camera Stream Support** | The system supports both local video files for testing and live network camera streams (RTSP) for real deployment. If a live stream drops due to a network issue, the system automatically attempts to reconnect rather than stopping entirely. |
