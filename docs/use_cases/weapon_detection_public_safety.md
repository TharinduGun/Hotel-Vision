# Weapon Detection for Public Safety Module — Use Case Summary

## What This Module Is

This module uses live or recorded CCTV footage to automatically detect weapons — handguns, rifles, shotguns, and knives — in real time. It is designed for hotels, public venues, and commercial buildings where identifying an armed individual as early as possible is critical for guest and staff safety. No human needs to monitor the camera feed continuously; the system watches every frame and raises alerts automatically when a weapon is detected with high confidence.

## Scope

The module sits between a camera and an alert system. It takes in video, processes it frame by frame, scans for weapons within person bounding boxes, validates detections through multiple filtering layers, and generates severity-rated alert events. Those events can trigger snapshots, video clips, and real-time notifications through the connected dashboard.

## What is Completed

| Feature | Description |
|---|---|
| **Weapon Detection** | The system can identify handguns, rifles, shotguns, and knives from a camera feed using a custom-trained YOLOv8 model. It works on both recorded video files and live camera streams. The model was fine-tuned with hard negative mining using real hotel lobby footage to reduce false positives from everyday objects like phones, wallets, and remote controls that can resemble weapons. |
| **Person-ROI Gated Detection** | Standard full-frame weapon detection scans the entire image and frequently mistakes background objects for weapons. The system addresses this by only scanning within the bounding box of each tracked person, expanded by a 30% padding margin for context. This dramatically reduces false positives because the model only evaluates areas where a person is actually present, and it saves GPU processing time by ignoring empty regions of the frame. |
| **Hand Proximity Filtering** | A weapon detection alone is not sufficient — the system also checks whether the detected weapon is near a person's hands. It uses a separate YOLOv8-pose model to locate wrist keypoints for every person in the frame. A detection is only accepted if the weapon's center is within 40% of the person's height from any detected wrist. This filters out objects near a person's torso, feet, or background that might look like weapons but are not being carried. |
| **Bounding Box Size Validation** | The system rejects detections that are geometrically implausible. If a weapon bounding box covers more than 40% of the associated person's body area, it is discarded — real weapons are small relative to a person. Detections smaller than 900 pixels (approximately 30×30) are also rejected as noise. Weapons shorter than 5% of the person's height are filtered out, and extreme aspect ratios beyond 5:1 are discarded. |
| **Temporal Consistency** | A single-frame detection is not treated as a confirmed weapon. The system requires a weapon to be detected in at least 3 out of the last 5 consecutive frames for the same person before raising an alert. This sliding-window approach eliminates one-off false positives caused by motion blur, glare, shadows, or brief occlusions, ensuring that only persistent, real detections trigger an alert. |
| **Per-Person Alert Cooldown** | Once an alert is raised for a specific person, the system suppresses further alerts for that same person for a configurable cooldown period (default 10 seconds). This prevents alert spam when a weapon is continuously visible — security staff get one clear notification and are not overwhelmed by repeated alerts every frame. |
| **Severity Classification** | Each detection is automatically assigned a severity level based on the weapon type. Handguns, rifles, and shotguns are rated as CRITICAL. Knives are rated as HIGH. This allows the alert system to prioritise its response — a firearm detection can trigger an immediate lockdown procedure while a knife detection might trigger a different response protocol. |
| **Snapshot & Clip Capture** | When a weapon is detected, the system automatically saves a high-resolution snapshot of the frame showing the detection bounding box. It also captures a short video clip (default 5 seconds) around the detection moment. These are stored locally and can be used as evidence or for post-incident review without needing to scrub through hours of raw footage. |
| **Person Tracking Integration** | Every weapon detection is linked to a specific tracked person via their track ID. If the camera briefly loses sight of a person — due to glare, shadow, or a brief occlusion — the tracker holds the ID for a short period rather than immediately treating them as a new person when they reappear. This means the cooldown and temporal consistency filters work correctly even through brief visual interruptions. |
| **Data Logging** | All weapon detection events are written to a structured data file. Each event records the module name, camera ID, timestamp, event type, confidence score, bounding box coordinates, severity level, frame index, person ID, weapon class, and detection mode (person-ROI or full-frame). This file can be opened in Excel or used by any reporting tool. |
| **Camera Stream Support** | The system supports both local video files for testing and live network camera streams (RTSP) for real deployment. If a live stream drops due to a network issue, the system automatically attempts to reconnect rather than stopping entirely. |
